P3 — 优化建议

P3 无 Error Boundary
Graph3DCanvas 崩溃（如 WebGL context lost）会导致整页白屏。建议包裹 ErrorBoundary 显示降级 UI。
P3 无可访问性
按钮无 aria-label、3D 画布无键盘导航、无 focus 管理。开源项目应考虑基础 a11y。
P3 JSON 导入校验过浅
只检查 nodes/edges 是 Array，不检查 id 唯一性、weight 范围、edge.source/target 是否存在于 nodes 中。恶意/错误 JSON 可能导致渲染异常。
P3 版本号不一致
package.json "version": "0.1.0"，UI 显示 v0.1，git tag 有 v0.1 / v0.1.1 / v0.1.2。应统一从 package.json 读取。

四、未来产品展望

核心判断

这个项目的护城河不是技术复杂度，而是"视觉冲击 × 话题性 × 一行嵌入"的传播三角。当前 MVP 已验证了视觉表现力，下一步应该围绕降低嵌入门槛和扩大话题覆盖两个轴展开，而不是继续堆功能。

Phase 1
v0.2

工程化加固 + npm 封装

拆分 God 组件、清理死代码、补类型定义、提取 <RelationGraph3D data={} /> 为独立 npm 包（@your/relation-graph-3d）。这是开源传播的基础——别人 npm install 后一行能用，才会 Star。

Phase 2
v0.3

话题性数据集矩阵

心理学图谱是好的开始，但传播需要多个话题入口：三国人物关系、漫威宇宙、硅谷公司投资链、《三体》人物网络、Linux 内核模块依赖。每个数据集就是一个社交媒体话题。数据集改为 JSON 异步加载，支持 URL 参数直接指定 ?dataset=sanguo。

Phase 3
v0.4

README 首屏动图 + 在线 Demo

README 的第一屏放一个 10 秒循环 WebM（三国图谱旋转+点击高亮+星云隐藏），比任何文字描述都有说服力。部署到 GitHub Pages，URL 分享链路打通。这是 Star 转化的单一最重要资产。

Phase 4
v0.5

性能优化第二波

力导向迁移 Web Worker（不阻塞 UI）、InstancedMesh/Sprite 按需更新（静止帧跳过）、可选 Bloom 后处理（用 postprocessing 库替代当前 fake glow）、LOD 距离自适应。目标：500 节点流畅 60fps。

Phase 5
v1.0

生态扩展

Vue 组件封装（vue-relation-graph-3d）、iframe 嵌入模式（给非前端用户）、CSV/Excel 导入（给数据分析师）、主题切换（亮色/暗色/赛博朋克）。此时项目已具备开源社区项目的完整形态。

不建议做的事

不要加后端。纯静态部署是核心卖点，一旦需要服务器就丧失了 GitHub Pages / Cloudflare Pages 的零成本传播优势。
不要加在线编辑器。这是可视化工具，不是 IDE。编辑需求交给 JSON 导入。
不要过早抽象插件系统。当前用户量为 0，插件系统是过度设计。
